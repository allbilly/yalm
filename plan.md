# Kernel comparison & profiling plan

Compare CUDA kernels across **yalm**, **calm**, **llama.cpp**, **tinygrad**, and **transformers** on Mistral-7B-Instruct-v0.2 fp16 decode, then profile bottlenecks and close the gap to bandwidth-bound “speed of light.”

References: [Fast LLM Inference From Scratch](https://andrewkchan.dev/posts/yalm.html) (`blog/blog.md`), `mistral_bench.py`, `ref/calm/`.

---

## 1. Baseline (this box, RTX 3080, 120 decode tokens)

| Engine | tok/s | ~% peak BW (760 GB/s) | Notes |
|--------|------:|----------------------:|-------|
| calm | **47.42** (10-run avg) | 90.3 % | Pre-built `.calm` weights |
| llama.cpp | **46.62** (10-run avg) | 88.8 % | FP16 GGUF |
| yalm (post-fix) | **45.82** (10-run avg) | 87.2 % | Hand-written CUDA in `src/infer.cu`; **+6.4 %** from §11 vectorization fix |
| transformers | 37.91 (1 run) | 72.2 % | PyTorch + cuBLAS / SDPA |
| tinygrad | **33.64** (10-run avg) | 64.1 % | Autogenerated kernels, `BEAM=8` |

Re-run: `pip install -r requirements.txt && .venv/bin/python mistral_bench.py`

**Speed of light** (blog §1.2): decode is memory-bandwidth-bound. Each token must read ~all weights + growing KV cache.

```
tok/s_max ≈ peak_DRAM_GB/s / bytes_per_token
bytes_per_token ≈ model_weights + KV_read_per_layer × layers
```

Mistral-7B fp16 weights ≈ **14.48 GB**. At 760 GB/s → **~52.5 tok/s** upper bound if decode were pure weight streaming (KV adds overhead; long context lowers effective ceiling).

**BW used** (from `mistral_bench.py`): `14.48 GB × tok/s / 760 GB/s`. Engines above ~90% are near the roofline; below ~75% have kernel or framework overhead worth profiling.

## 2. How each engine generates kernels

### 2.1 yalm — explicit, fused, graph-captured

**Source:** `src/infer.cu` (~1.2k lines), launched via CUDA graphs in the block forward pass.

| Kernel / fusion | Role |
|-----------------|------|
| `matmul` / `matmul_wide` | FP16 weight × FP32 activation GEMV; `matmul_wide` coalesces warp writes via `blocktranspose` |
| `fused_matmul_add_residuals` | Matmul + residual in one pass |
| `fused_qkv_matmul_clip` | Q/K/V projection + clip |
| `fused_rope_and_cache_update` | RoPE + KV write |
| `attn_dot` / `attn_softmax` | Attention scores |
| `att_mix` | Value mix (historically the worst kernel — blog §3.4) |
| `fused_ffn_w1_w3_glu_act` | GLU FFN gate + up + activation |

**Philosophy:** One readable kernel per logical op, then **fuse** what shares the same output shape. Blog found **~94% of GPU time in matmul-class work** after naive port; fusion + coalescing closed most of the gap to llama.cpp.

**Isolated bench:** `make test && ./build/test -bk matmul|mha|ffn` (Mistral-shaped dims in `src/test.cpp`).

### 2.2 calm — one cooperative kernel per forward

**Source:** `ref/calm/src/infer.cu` — `kernel_forward` launched with `cudaLaunchCooperativeKernel` (1024 threads/SM, persistent grid).

| Piece | Role |
|-------|------|
| `kernel_embed` | Token lookup |
| `kernel_forward` | Entire transformer layer: RMSNorm → QKV+RoPE+KV → attn score → softmax → mix → proj → FFN → residual, inside one grid |
| `kernel_output` | Final norm + LM head |
| KV layout | Keys tiled, values transposed for `attn_score` / mix locality |

**Philosophy:** Minimize launch overhead and global traffic by keeping work on-SM. Uses **dynamic parallelism patterns** and careful memory layout (see comments at lines 474–479 in `infer.cu`). Ships `tools/cudaprof.cu` (CUPTI) for per-kernel timing and peak BW per launch.

**Why faster than yalm on 3080:** Less kernel launch tax, better KV access patterns, highly tuned warppar matmul inside `kernel_forward`.

### 2.3 llama.cpp — GGML CUDA template zoo

**Source:** `~/llama.cpp/ggml/src/ggml-cuda/` — not vendored in yalm repo.

| Family | Role |
|--------|------|
| `mmvq` / `mmq` | Matrix–vector / matrix–matrix quantized mul (`mmq.cuh` + template instances per dtype) |
| `fattn-*` | Flash-attention-style kernels when enabled |
| cuBLAS fallback | Larger batches / prefill |

**Philosophy:** Hand-tuned **quantized** GEMV/GEMM for decode (even fp16 models use optimized paths). Massive instantiation matrix (q4_K, q8_0, …). Mature attention kernels for long context.

**Profile:** `nsys profile ~/llama.cpp/build/bin/llama-cli ...` — expect `mul_mat_vec_*` and `fattn_*` to dominate decode.

### 2.4 tinygrad — compiler-generated, BEAM-tuned

**Source:** `tinygrad_mistral.py` (vendored Llama) + pip `tinygrad` renderer.

| Piece | Role |
|-------|------|
| `scaled_dot_product_attention` | Generic attention (not Flash Attention on Mistral path) |
| Autogen CUDA | `BEAM=N` searches LOCAL/UPCAST configs at compile time |
| `TinyJit` | Reuses compiled graph for single-token decode |

**Philosophy:** No hand-written Mistral kernels. Speed comes from **search**, not fusion expertise. Trails native engines because decode is bandwidth-bound and generic matmul/attention underfill DRAM.

**Inspect generated code:** `DEBUG=4 BEAM=8 .venv/bin/python tinygrad_mistral.py --count 1 2>&1 | less` — look for `r_` kernel names and achieved GB/s in logs.

### 2.5 transformers — library stack

**Source:** PyTorch ops → cuBLAS/cuDNN + `scaled_dot_product_attention` (may pick flash/mem-efficient backends).

**Philosophy:** General framework; overhead from Python, extra buffers, unfused op boundaries. Good correctness baseline, not a kernel comparison target.

## 3. Decode-step kernel timeline (conceptual)

One token, one layer — what each engine *roughly* executes:

```
yalm:       [qkv_fused] → [rope+kv] → [attn_dot] → [softmax] → [att_mix] → [proj_fused] → [ffn_fused]   (×32 layers, graph)
calm:       [kernel_forward: all of the above inside cooperative grid]                                    (×32 launches)
llama.cpp:  [ggml_mul_mat_vec] × N  +  [fattn or vec attn]  +  [ggml GLU ops]                          (×layers)
tinygrad:   [rendered matmul/elem] + [sdpa] + …                                                         (×layers, JIT)
```

**Prefill vs decode:** This plan focuses on **decode** (1 token/step) — the bench prompt is short; prefill is negligible. Re-profile at **kv_len ≈ 4096** to stress attention (blog §3.4: `att_mix` blew up at long context).

## 4. Profiling workflow (from the blog)

The blog uses a two-tool loop: **Nsight Systems** for *where time goes*, **Nsight Compute** for *why a kernel is slow*.

### 4.1 Nsight Systems (`nsys`) — system timeline

**When:** First pass on full generation. Answers: GPU idle gaps? CPU bound? Which kernels dominate?

```bash
# yalm (from README)
nsys profile --trace=cuda,nvtx,osrt --stats=true \
  ./build/main mistral-7b-instruct-fp16.yalm -d cuda -m completion \
  -i "Q: What is the meaning of life?" -n 120 > profile.out

# calm
nsys profile --trace=cuda --stats=true \
  ~/calm/build/run ~/.cache/mistral-7b-instruct.fp16.calm -c 4096 -n 120 \
  -i "Q: What is the meaning of life?"

# llama.cpp
nsys profile --trace=cuda --stats=true \
  ~/llama.cpp/build/bin/llama-cli -m ~/mistral-7b-instruct-v0.2.fp16.gguf \
  -c 4096 -n 120 -p "Q: What is the meaning of life?" --no-display-prompt --single-turn

# tinygrad (noisier — includes compile on first run; warmup first)
BEAM=8 .venv/bin/python tinygrad_mistral.py --count 120
nsys profile --trace=cuda --stats=true \
  BEAM=8 .venv/bin/python tinygrad_mistral.py --count 120
```

**Read the report:**

1. **CUDA HW row** — gaps → launch overhead, sync, or CPU. Blog: yalm gaps were µs-scale (not CPU-bound).
2. **CUDA Kernel Statistics** — sort by total time. Blog: **94.4% matmul** before fusion.
3. **Compare engines** — calm should show fewer, longer kernels; yalm many small kernels; tinygrad many `r_*` kernels.

### 4.2 Nsight Compute (`ncu`) — kernel autopsy

**When:** A specific kernel is >5% of time *or* nsys shows low DRAM throughput. Answers: coalescing, occupancy, memory % of peak, compiler behavior.

```bash
# yalm isolated kernels (no full model) — best for A/B kernel tweaks
sudo ncu --set full -o report_matmul ./build/test -bk matmul
sudo ncu --set full -o report_mha   ./build/test -bk mha

# Single kernel from full run (replace kernel name from nsys)
sudo ncu --kernel-name regex:att_mix --launch-count 20 \
  ./build/main mistral-7b-instruct-fp16.yalm -d cuda -m completion -i "Q: hi" -n 10
```

**Key metrics (blog §3.3–3.5):**

| Metric / warning | Meaning | Action |
|------------------|---------|--------|
| **DRAM Throughput %** | % of chip peak BW | <60% on a memory-bound kernel → layout/coalescing bug |
| **Uncoalesced stores** (L1TEX→L2) | Warp writes not merged to 128 B lines | `matmul_wide` / `blocktranspose` pattern |
| **Compute Throughput %** | ALU use | High compute + low DRAM on GEMV → probably fine; low both → latency stalls |
| **Warp State (Stall)** | Memory throttle, barrier, not selected | Guides occupancy / sync fixes |
| **SASS source view** (`-set full`) | PTX/SASS per source line | Blog: FP32 `att_mix` got loop unroll + prefetch; FP16 did not → manual unroll |

### 4.3 calm built-in profiler

```bash
# Build calm with profiling enabled (see ref/calm/tools/cudaprof.cu)
# Reports per-kernel time %, avg µs, peak BW, SM occupancy
```

Use to compare calm’s internal breakdown against yalm’s nsys slice.

### 4.4 Memory bandwidth sanity check

Establish **achievable** peak on this GPU before blaming kernels:

```bash
./build/test -b    # single-threaded host BW
./build/test -b2   # multithreaded host BW
```

Blog uses measured peak to interpret `Max Bandwidth %` in ncu. RTX 3080 spec: 760 GB/s; calm notes practical ceiling ~5% below spec on some cards.

## 5. Bottleneck identification checklist

Work top-down:

```
1. End-to-end tok/s  →  mistral_bench.py
        ↓
2. % peak BW         →  if <85% on decode, continue
        ↓
3. nsys timeline     →  GPU idle? kernel count? top 3 kernels by time?
        ↓
4. Classify dominant kernel type:
        ├─ matmul / GEMV     → coalescing, fusion, FP16 loads, tensor core path
        ├─ att_mix / attention → KV layout, atomic patterns, seq_len scaling
        ├─ elementwise/norm  → fuse into neighbor matmul
        └─ launch/sync       → CUDA graphs (yalm), cooperative kernel (calm)
        ↓
5. ncu on hot kernel   → DRAM%, uncoalesced, occupancy, SASS
        ↓
6. Isolated -bk test   → verify fix without full model noise
        ↓
7. Re-bench tok/s + long-context (n=4800) regression
```

**Expected dominant ops (Mistral 7B, short decode):** matmul-class (QKV, proj, FFN) ≫ attention. At **kv_len→4096**, attention share rises; blog expected FFN ≈ 3.5× attn scoring kernel time — if `att_mix` ≈ FFN, it’s wrong.

**Red flags from blog (apply to any engine):**

- `att_mix` at ~8% DRAM throughput → memory access pattern bug, not “attention is hard”
- `atomicAdd` to global for tiny floats → subnormals flushed to zero (quality + perf)
- Naive FP16 KV → smaller transactions, lost compiler unroll → **slower** than FP32
- One warp per output element writes → uncoalesced stores

Phases A–E executed end-to-end. Full per-kernel data, source diff, and code
patches are in `results.md`. This section is the summary against the plan.

### 11.1 Phase A — baselines reproduced

**Short context (120 tok, 10-run mean except transformers)** — see §1 table
above. yalm **43.08 → 45.82 tok/s (+6.4 %)** from the §11.4 fix.

**Long context (4100-tok prompt, 30 decode steps):**

| Engine        | tok/s at ~4k ctx | vs. short    |
|---------------|-----------------:|-------------:|
| llama.cpp     |           46.5   | -0.12        |
| calm          |           47.4   | +0.0 (flat)  |
| yalm (before) |           35.9   | -7.2 (-17 %) |
| yalm (after)  |       **41.85**  | -3.97 (-9 %) |

`att_mix` at full kv_len=4096 is **6.6 ms × 32 layers = 213 ms/token** — that
is ~10× yalm's per-token budget. This is the **structural** long-context
problem: V cache layout `(seq_len, n_kv_heads, head_dim)` → strided access
for `att_mix`; calm transposes to `(n_kv_heads, head_dim, seq_len)` →
contiguous.

`ncu` perf counters require root in this env (ERR_NVGPUCTRPERM), so the
detailed warp-stall / occupancy / coalescing analysis was done via
arithmetic on these `-bk` numbers and the nsys data, not directly.

### 11.4 Phase E — fix applied and verified

**Change** (one file, `src/infer.cu` lines 215-251, ~30 lines):
`matmul_row(half*, float*, …)` and `matmul_row(float*, float*, …)` were doing
scalar `row[j]`/`x[j]` loads with `j = offset, offset+32, …` — 32 separate
2 B/4 B transactions per warp per iter. Replaced with `half2`/`float2` paired
loads at `j = offset, offset+32, …` over the **half2 array** so each iter is
one 128 B (row) + one 256 B (x) coalesced sector — half the iterations, full
transaction width. This is the same coalesced-load pattern calm uses in
`matmul_warppar` (helpers.cuh:127-137).

Called by: `matmul`, `matmul_wide`, `fused_matmul_add_residuals`,
`fused_qkv_matmul_clip`, `fused_ffn_w1_w3_glu_act` — 5 of the 6 matmul-class
kernels in yalm.

**Verification:**

### 11.5 Source-level gap analysis (Phase D summary)

What was found vs. what was fixed:

| # | Gap                                                            | Impact | Status |
|--:|---------------------------------------------------------------|-------:|--------|
| 1 | `matmul_row` scalar loads (strided 64 B per lane)            |  62 % of GPU time affected | **Fixed** (this run) |

### 11.6 Prioritized next-fix list

| # | Fix                                                        | Expected gain      | Difficulty | Why |
|--:|------------------------------------------------------------|-------------------:|-----------:|-----|
| 1 | Coalesce FFN GLU writes (`blocktranspose` pattern)        | +3–5 % short       | medium     | 50 % of GPU time; same pattern already used by `fused_matmul_add_residuals` |
| 2 | Coalesce QKV clip writes (same pattern)                    | +1–2 % short       | medium     | 11 % of GPU time; 3 outputs per warp may need 3 blocktransposes |
| 3 | Transpose V cache for `att_mix` (calm-style)               | +5–10 % at kv_len=4k | high     | long-context only; requires `fused_rope_and_cache_update` update |
| 4 | FlashAttention for the attn triple                         | +5–15 % long-context | high      | structural; replaces 3 kernels with 1; matches llama.cpp |

**#1** closes yalm's short-context gap to calm from 1.6 tok/s to ~0.5 tok/s.
**#3** closes the long-context gap. Both are recommended.

### 11.7 Caveats and follow-ups

- The `mistral_bench.py` harness OOMs on this 20 GB card when `transformers`
  and `yalm`/`calm`/`llama.cpp` run in the same Python process. The harness
  was not modified; engines were run as separate subprocesses.
- `ncu` perf counters require root (no sudo in this env). SASS / warp-stall
  / occupancy was not run directly. All kernel-level numbers in this report
  are from `nsys` and arithmetic on the `-bk` tests.
- calm's cooperative kernel structure means nsys can only see it as a single
  `kernel_forward` time. Per-kernel comparison is not possible for calm —
  only totals.
- Plan said long-context attention rises to ~10 % of runtime for yalm.
  It's actually worse: `att_mix` is 0.6 % at short but 6.6 ms/iter isolated
  → ~213 ms/token at full kv_len ≈ 90 % of per-token budget. Confirms the
  priority of #3 in §11.6.
